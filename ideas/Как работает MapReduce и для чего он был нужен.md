# Как работает MapReduce и для чего он был нужен

**Пример задачи и** **как ее можно сделать без MapReduce.**

Задача: на входе терабайты текста (как правило, на разных машинах) и вам нужно посчитать, сколько раз в тексте встречается каждое слово. Считать на одной машине - уйдет много времени. Вы захотите распараллелить и, допустим, решаете выделить одну машину на каждые 100гб текста. Каждая машина считает, сколько раз встречалось каждое слово в данных ей 100гб текста. Затем оказывается, что одно и то же слово, например "apple", со своим счетчиком оказывается на нескольких посчитавших машинах. Нужно теперь как-то отправить все счетчики для слова "apple" на одну машину, чтобы там их просуммировать. И для остальных слов отправить их счетчики так же на ту же единую машину. Получается бутылочное горлышко в виде этой одной машины. Можно сделать иначе - завести R штук таких машин и для каждого слова однозначно определять ( hash(слово) mod R ), в какую из R машин оно отправится. Каждая из машин создаст свой файл со строчками слово, кол-во, и у нас по итогу на выходе R файлов, не пересекающихся по словам. 

## Как задача решалась бы с MapReduce

В MapReduce под капотом используется та же идея, что и в ручном распараллеливании, но все детали передачи данных, отказоустойчивости и синхронизации берет на себя фреймворк. Разработчик описывает только логику вычислений через две функции: `map` и `reduce`.

Рассмотрим тот же пример с подсчетом слов.

### Функция map

Функция `map` по умолчанию принимает на вход пару:
<имя_файла, содержимое_файла>

Разработчик описывает логику следующим образом:
- распарсить содержимое файла
- для каждого найденного слова сгенерировать пару: <слово, 1>

Важно: функция `map` **не знает**, сколько будет редьюсеров, где они находятся и как данные будут передаваться. Она просто эмитит пары `<key, value>`.

В общем виде: map(inputKey, inputValue) -> список пар <key, value>

В нашем случае:
- `key` — слово
- `value` — число `1`

Тип ключа и значения задается явно (например, `Text`, `IntWritable` в Hadoop), и именно это позволяет фреймворку:
- сериализовать ключ
- вычислять для него хэш
- сравнивать и сортировать ключи

---

### Shuffle, partition и sort (внутренняя часть фреймворка)

После выполнения `map` фреймворк берет все сгенерированные пары `<key, value>` и выполняет следующие шаги:

1. **Partition**  
   Для каждой пары определяется, какому reducer’у она будет отправлена.  
   По умолчанию используется формула: partition = hash(key) mod R

где `R` — количество reducer’ов.

Это гарантирует, что **все пары с одинаковым ключом попадут к одному и тому же reducer’у**.

При необходимости разработчик может задать собственный `partitioner`, определяющий, как именно ключи распределяются по партициям.

2. **Shuffle**  
Map-задачи записывают данные локально на диск, разбивая их на `R` файлов (по одному на каждую партицию), и сообщают центральному узлу (master), где эти файлы находятся.

3. **Sort & Group**  
Перед передачей в reduce данные:
- сортируются по ключу
- группируются так, что reducer получает: <key, список всех value для этого key>

---

### Функция reduce

После завершения всех map-задач фреймворк запускает `R` reduce-задач — по одной на каждую партицию.

Каждая reduce-задача:
- читает свою партицию со всех машин, где выполнялись map-задачи
- получает данные в виде: <слово, [1, 1, 1, …]>
- применяет пользовательскую функцию `reduce`

В нашем примере `reduce` просто суммирует значения: reduce(слово, список_единичек) -> сумма

Результат записывается в файл, соответствующий данной партиции. В итоге на выходе получается `R` файлов, не пересекающихся по ключам (словам).

В общем виде: reduce(key, список value) -> один value или несколько value

---

### Итог

- `map` отвечает только за генерацию пар `<key, value>`
- фреймворк автоматически выполняет:
  - partition
  - shuffle
  - sort
  - group
- `reduce` работает с уже сгруппированными данными

MapReduce был придуман в Google, чтобы избавить разработчика от необходимости вручную управлять распределенными вычислениями и позволить сосредоточиться исключительно на логике обработки данных.

**Какая есть проблема с MapReduce**

Введем два понятия:

_Детерминированная функция_ - для одних и тех же входных данных разные запуски вернут один и тот же результат.

_Недетерминированная функция_ - для одних и тех же входных данные разные запуски функции могут вернуть разный результат. Банальный пример: в коде функции есть вызов random().

Если map-функция детерминирована, то проблем нет. Разработчик пишет реализацию map и reduce и представляет, как будто его программа будет выполняться не параллельно на нескольких машинах, а последовательно на одной. Так писать код намного проще.

Если же map-функция недетерминирована (может сгенерировать разные key на разных запусках), то начинаются проблемы. 

Пусть была map-задача M1, которая на одной из машин прогнала map-функцию по куску входных данных. Были сгенерированы R файлов-партиций с парами <key, value>: R1, R2, R3.

Спустя какое-то время приходит reduce-задача, вычитывает R1 (свою партицию) и успевает завершиться, пока две другие reduce-задачи не успели даже вычитать свои партиции.

И в этот момент машина с М1 крашится. Теперь нужно перезапускать М1 на другой машине, чтобы две оставшиеся reduce-задачи дочитали свои партиции.

Только вот проблема - недетерминированная map-функция нагенерирует теперь совсем другие key, данные раскидаются по партициям совершенно иначе.

Разработчику уже не получается представить, будто его программа выполняется последовательно на одной машине.

Если кратко, то проблема формулируется так: **недетерминированная map-функция может приводить к нежелательными и неожиданным результатам**.

Собственно, **способы пофиксить это**:

1. Не писать недетерминированные map-фукнции😄. Не очень реалистично, от такого не защитишься.

2. Перезапускать все reduce-задачи в случае фейла любой map-задачи - не вариант, джобы будут работать вечность.

3. Писать партиции не на локальный диск, а в распределенную файловую систему с репликацией - сильно замедлит работу. Впрочем, в Google в т.ч. по этой причине изобрели Colossus - более продвинутую и быструю распределенную файловую систему на смену устаревшей Google File System (GFS).
