# Применение HBase для Facebook Messanger

HBase - это open-source копипаста Big table. Между ними можно ставить знак равенства.

Facebook messenger 8 лет хранил все свои сообщения в HBase.

В этой заметке опишу, почему в HBase и какая была схема данных. Схема данных позволит прочувствовать, как можно использовать wide column базы.

  

**Почему в HBase**

Несколько вводных:

- в большинстве бд используются индексы на основе b-дерева (сбалансированное дерево поиска на диске)

- в HBase в качестве индекса используются LSM-деревья

- структура сообщения выглядит примерно так:

```
struct message {

  string chatId

  int64 time

  string messageText

  string senderId

}

```

А теперь представим, что мы используем какую-нибудь привычную базу данных с b-деревом в кач-ве индекса.

Юзеров и сообщений так много, что одна машина не выдержит такой поток записей (да и чтений). Потому нужно шардировать.

Канонически будем шардировать по chatId - все сообщения для одного чата оказываются на одной машине, что позволяет, зайдя в чат и листая историю, читать из одной машины, а не мержить кучу сообщений с разных машин.

Одна машина обрабатывает запросы по нескольким чатам, большому их кол-ву.

На машине нужно уметь быстро доставать сообщения по chatId, т.е. нужен индекс по chatId. Т.е. имеем b-дерево, посортированное по chatId. Узлы дерева живут на разных участках диска.

И вот тут начинается проблема: на одну машину постоянно прилетают сообщения для разных chatId -> происходят записи в разные участки на диске -> постоянные disk seek-и при записи. Производительность убивается напрочь.

А теперь представим, что используем HBase.

Напомню модель данных.

По ключу имеем сколько угодно колонок и в каждой колонке для ключа имеем историю значений timestamp,value. (Можно настраивать, сколько последних timestamp-ов храним).


```
column1            column2

key1      t10,val10          ...

          t9,val9

key2      ...                ...

key3      ...                ...

```
Хранятся данные в SSTables. Sorted string tables - это просто строковые файлы, где каждая строка в формате key,value.

А конкретно, в SSTables лежат вот какие key и value:

```
key1:column1:t10,val10

key1:column1:t9,val9

...

key3:column3:t6,someOtherVal
```
И сортировка идет по составному key - по тому, что до запятой. Т.е. **ключом в SSTable становится комбинация из изначального key, колонки и timestamp**.

И, например, когда хотим узнать значения по key1 в колонке column2, то бинпоиском находим в файле нужный диапазон строк.

Если хотим уточнить одно значение для какого-то timestamp-a, то так же бинпоиском находим конкретную строку.

Что мы видим: значения из одной ячейки хранятся последовательно в файле. Просто все значения для одного key - тоже.

  

Напомню, как получается отсортированность.

Все апдейты попадают в mem-table, дерево поиска в памяти. Поддерживать отсортированность по keyX:columnX:timestampX в памяти элементарно. И писать разросшийся mem-table на диск, сохраняя отсортированность - тоже элементарно. Причем запись на диск, как было упомянуто в одной из заметок выше, делается на диск последовательно - ведь просто пишем в один файлик в конец.

И вот тут-то и кроется причина, почему в Facebook заюзали HBase для сообщений. Новые сообщения не приводят к апдейтам в разных местах на диске, как в случае с b-деревом. Все апдейты пишутся последовательно в конец SSTable-файла.

**Так какая же модель данных для хранения сообщений?**

```
chatId1                        chatId2

userId1   t10,messageText&senderId       ...

          t9,messageText&senderId

userId2      ...                         ...

userId3      ...                         ...
```
В качестве ключа - userId. (Это может показаться странным, но да, по ключу userId лежат все сообщения, которые может прочитать userId - и отправленные самим userId, и отправленные ему кем-то. Т.е. одно сообщение копируетяся для отправителя и получателя или нескольких получателей. Такое дублирование даже полезно - так можно удобнее разделить статусы отправки и получения)

И да, для каждого чата - отдельная колонка🙈Все-таки wide column - это интересно.

В кач-ве timestamp используем время отправки сообщения.

Что мы получаем на диске в SSTables:

```
userId1:chatId1:t10,messageText&senderId

userId1:chatId1:t9,messageText&senderId

...

userId3:chatId123:t6,messageText&senderId
```

Что это дает?

Для одного юзера все относящиеся к нему сообщения рядом на диске. В рамках одного чата - еще "более рядом". Супер-эффективное чтение сразу "диапазонов" сообщений, а это ведь основной паттерн доступ - мы заходим в чат и листаем, видим соседние сообщения. Хотим прочитать сообщения из чата - просто делаем lookup по userId,chatId, а под капотом бинпоиском находим смежные записи в SSTable.

Ну и, повторюсь, новые сообщения пишутся на диск последовательно в один файл, без disk seek-ов.

Все это работает намного лучше привычных баз данных с b-деревом в кач-ве индекса.

**А как же** **еще запись в Write ahead log**

Напомню, что mem-table можно потерять, если машина упадет, а потому перед апдейтом mem-table мы пишем данные об этом новом апдейте на диск в write ahead log.

И у нас таких write ahead log-ов - огромное кол-во на одну машину, ведь tablet-ов обслуживается на машине много (по lsm-дереву на tablet).

Не получаем ли мы такую же проблему, как с b-деревом?

В Google придумали, как это обойти - сделали единый write ahead log для всех tablet-ов (было тут, см пункт 4).

Итого, на одной машине у нас идет _постоянная_ _и последовательная_ запись в один write ahead log-файл и лишь периодическая в новые SSTables.

Это намного лучше, чем для b-дерева бегать по диску на запись каждого нового сообщения.
