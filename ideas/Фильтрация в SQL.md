# Фильтрация в SQL

**SELECT, WHERE и HAVING - как именно они работают**

WHERE и HAVING - это **фильтрация** строк.

SELECT - только **проекция**, т.е. какие _колонки_ вернуть. И на самом деле запускается в конце.

  

```
FROM users

JOIN departments ON users.dep_id = departments.id
```
даст нам изначальный набор строк с людьми и их департаментами.

  
```
WHERE departments.type = 'engineering'
```
отфильтрует строки с инженерным департаментом.

А дальше мы хотим отфильтровать только строки с департаментом из >100 человек:

```
GROUP BY departments.id

HAVING COUNT(users.id) > 100
```
То есть HAVING - это фильтр как WHERE, только продвинутый: по свойствам группы, в которой оказалась строка. Свойство в данном примере - размер группы. Могут быть и другие - например, сумма по какому-то полю.

А в конце у нас SELECT, в котором мы указываем, какие колонки нам интересны:

```
SELECT departments.id, COUNT(user.id)
```
В SELECT мы обязаны положить либо все поля, по которым делали GROUP BY, либо агрегирующую ф-ию, либо и то, и другое.

Это потому, что на вход SELECT-у идут не строки, а _группы_ строк. И мы можем селектить либо "название" группы - поля из GROUP BY, либо "свойства" группы - агрегирующие ф-ии.

SELECT-у нужно однозначно понимать, как селектить. Если указать поле не из GROUP BY, то SELECT не поймет, из какой строки в группе брать поле.

**Почему в WHERE не стоит фильтровать по вычисленному полю, а в HAVING можно**

Т.е. так лучше не делать:

```
SELECT users.id

FROM users 

WHERE CONCAT(firstName, ' ', lastName) = '...'
```
А вот так норм:

```
SELECT registrationDate

FROM users

WHERE ...

GROUP BY registrationDate

HAVING COUNT(users.id) > 100 AND AVG(...) > 15
```

Дело в том, что фильтры в WHERE потенциально могут использовать индекс, который ускорит запрос. Если мы сделаем фильтр по вычисленному полю, то можем пойти мимо индекса.


В HAVING попадают все строки после WHERE и, чтобы посчитать группировку, нужно в любом случае прочитать все входящие строки. А потому можно делать фильтры по любым вычисляемым полям (COUNT, SUM, AVG и тд), по любой их комбинации, и не бояться замедления запроса.


**Немного оффтоп:**

Нельзя фильтровать по alias-ам из SELECT:

```
SELECT users.id, CONCAT(firstName, ' ', lastName) as fullName

FROM users 

WHERE fullName = '...'
```
, тк на этапе WHERE ничего не известно про SELECT и его alias-ы.

То же относится и к HAVING.

**Как под капотом работает GROUP BY & HAVING**

Когда мы делаем

```
GROUP BY key1, key2

HAVING COUNT(*) > 100
```

, происходит одно из двух:

1. либо заводится хэш-тиблица с ключом key1,key2.

2. либо все строки сортируются по key1,key2.
  

(Далее, для расчета агрегаций, в обоих случаях делается линейный проход по всем строкам)

Первый вариант предпочтителен, если уникальных комбинаций key1,key2 мало.

Второй вариант предпочтителен, если уже есть индекс по key1,key2 - не нужно сортировать.


Что выбирет планировщик, не всегда можно угадать, но если 

а) строк очень много, 

б) уникальных комбинаций ключей в GROUP BY также очень много, 

то, возможно, есть смысл завести индекс по ключам из GROUP BY.
