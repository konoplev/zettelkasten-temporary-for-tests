# Skip List

Скриптлист — это структура данных для хранения отсортированных элементов, похожая на односвязный список, но с иерархией узлов разной высоты, что позволяет «перепрыгивать» через диапазоны элементов. Она использует вероятностное балансирование вместо перестановок при вставках и обновлениях. Скриптлисты проще, чем деревья поиска, но менее эффективны для кэширования; некоторые реализации улучшают это через объединённые списки.

![[Pasted image 20260203160252.png]]

Скриптлист имеет несколько преимуществ перед отсортированным массивом и бинарным деревом поиска:
	1.	Быстрые вставки и удаления:
	•	В отсортированном массиве вставка или удаление требует сдвига элементов → O(n).
	•	В скриптлисте вставка и удаление работают за O(log n) в среднем, без сложных поворотов, как в сбалансированном дереве.
	2.	Простота реализации:
	•	Скриптлист проще, чем самобалансирующиеся деревья (AVL, Red-Black), потому что не требует сложных операций поворота.
	3.	Поддержка поиска диапазонов:
	•	Легко пройтись по элементам в диапазоне, так как структура уже поддерживает последовательные ссылки.
	4.	Гибкое балансирование:
	•	Балансировка происходит вероятностно, что делает структуру адаптивной без сложных алгоритмов.

Минусы есть: кэш-производительность (использование кэша CPU) ниже, чем у массивов или B-деревьев, и худший случай может быть O(n).

## Структура
Skiplist — это несколько уровней связанных списков:
	•	Нижний уровень: обычный отсортированный связанный список всех элементов.
	•	Верхние уровни: «пропускают» несколько элементов, соединяя узлы через интервалы.
	•	Узел: хранит ключ и ссылки на узлы ниже и справа (по высоте узла).
	•	Высота узла: выбирается случайно; чем выше, тем больше узлов он пропускает.

По сути, это лестница из связанных списков: чем выше уровень, тем длиннее «шаги» между узлами.
Первая нода - рут (в ней только ссылки), последняя - бесконечность. 

## Поиск

Начинают с верхнего уровня слева, движутся вправо, пока следующий узел не больше искомого, затем спускаются на уровень ниже, повторяя, пока не найдут элемент или не дойдут до конца.

Ищем 7

![[Pasted image 20260203160252.png]]
1. Следуем по указателю на верхнем уровне к узлу с ключом 10.
2. Поскольку искомый ключ 7 меньше 10, идём по указателю на следующий уровень от головного узла, находя узел с ключом 5.
3. Следуем по указателю верхнего уровня этого узла, снова находя узел с ключом 10.
4. Искомый ключ 7 меньше 10, поэтому идём по указателю на следующий уровень от узла с ключом 5, находя узел с искомым ключом 7.

## Вставка

При вставке точка вставки (узел с ключом или его предшественник) определяется с помощью описанного ранее алгоритма поиска, после чего создаётся новый узел. Чтобы построить иерархию, похожую на дерево, и сохранить баланс, высота узла выбирается случайным образом на основе вероятностного распределения. Указатели в предшествующих узлах с ключами, меньшими, чем ключ нового узла (Все узлы, с которых при спуске на уровень ниже нужно было «свернуть» вправо, и будут **предшествующими узлами на каждом уровне**, чьи указатели нужно перенаправить на новый узел), перенаправляются на этот новый узел, при этом их указатели на более высоких уровнях остаются без изменений. Указатели в новом узле связываются с соответствующими последующими узлами на каждом уровне.

## Удаление

При удалении указатели удаляемого узла на следующих уровнях перенаправляются на предшествующие узлы на соответствующих уровнях.

Перед удалением алгоритм сначала ищет ключ, как при обычном поиске. Во время этого поиска:
	•	Предшествующие узлы запоминаются на каждом уровне — это узлы, чьи указатели нужно будет перенаправить.
	•	Следующие узлы естественно находятся как узлы справа от предшествующих на каждом уровне — на них будут указывать эти перенаправленные указатели.

То есть, оба типа узлов определяются в процессе поиска точки удаления, дополнительных обходов не требуется.

## Concurrency
Можно создать конкурентную версию скиплиста, реализовав схему линейной согласованности, которая использует дополнительный флаг fully_linked, определяющий, полностью ли обновлены указатели узла. Этот флаг можно устанавливать с помощью операции compare-and-swap. Это необходимо, потому что указатели узла должны быть обновлены на нескольких уровнях, чтобы полностью восстановить структуру скиплиста.

Цель: вставить узел в скиплист, чтобы другие потоки видели его линейно согласованно.

Шаги:
	1.	Поиск позиции вставки
	•	Используем обычный алгоритм поиска в скиплисте.
	•	Находим массив предшествующих узлов (predecessors) и массив следующих узлов (successors) для всех уровней, на которых будет новый узел.
	2.	Создание нового узла
	•	Выбираем случайную высоту узла по вероятностной схеме.
	•	Создаём новый узел с указателями на successors на всех уровнях.
	3.	Установка указателей сверху вниз
	•	Для каждого уровня от нижнего к верхнему:
	•	Пытаемся перенаправить указатель предшествующего узла на новый узел с помощью CAS (compare-and-swap).
	•	Если CAS неудачен, повторяем попытку (другой поток мог изменить указатель).
	4.	Установка флага fully_linked
	•	После успешного обновления указателей на всех уровнях устанавливаем fully_linked = true.
	•	Только после этого узел считается полностью вставленным и доступным для других потоков.

Особенность: флаг fully_linked гарантирует, что пока узел не полностью вставлен, другие потоки его не используют.


В языках с неуправляемой моделью памяти можно использовать подсчёт ссылок или hazard pointers, чтобы гарантировать, что узлы, на которые в данный момент есть ссылки, не будут освобождены при одновременном доступе. Этот алгоритм свободен от взаимных блокировок, так как узлы всегда обрабатываются начиная с верхних уровней.

Проблема: если другой поток удаляет узел, на который есть ссылки, он может быть освобождён в памяти, что вызовет ошибку.

Решения:
	1.	Подсчёт ссылок (reference counting)
	•	Каждый узел хранит счётчик ссылок.
	•	Перед доступом поток увеличивает счётчик, после окончания работы — уменьшает.
	•	Узел освобождается только когда счётчик станет 0.
	2.	Hazard pointers
	•	Каждый поток указывает, на какие узлы он в данный момент ссылается (hazard pointer).
	•	Узлы, на которые есть активные hazard pointers, не удаляются.
	•	Поток, удаляющий узел, откладывает его освобождение, пока никто не ссылается.

Плюсы обоих методов:
	•	Гарантируют, что узел не будет удалён во время доступа.
	•	Алгоритм остаётся без deadlock, так как узлы всегда обрабатываются сверху вниз — потоки не блокируют друг друга в цикле.
