# Redis

Redis - это in memory key-value кэш.
Кэш отличается от просто хранилища тем, что автоматически делает cache evictions (извлечения из кэша), когда кэш становится переполненным. Предоставляет разные стратегии для этого: LRU, LFU, также по ключу можно задавать TTL.

Redis выполняет команды от всех клиентов в одном потоке. Причина - операции делаются в памяти и супер быстрые (десятки микросекунд), а потому нет смысла вносить параллелизм, который всё только усложнит.

Транзакции

Неполноценные, но хорошие. Redis однопоточный и потому транзакции реализованы элементарно: Redis выполняет все команды из транзакции друг за другом, не перемешивая с командами от других клиентов. Клиент просто говорит:
```
MULTI
…commands…
EXEC
```
Без MULTI EXEC команды могли бы перемешаться с другими от других клиентов.

**Атомарность транзакций в Redis обеспечивается полностью.**

Во-первых, чтения в одном потоке с записями, а потому никто прочитать посередине транзакции не сможет.

Во-вторых, если на диск в write ahead log записалась только часть операций, затем Redis упал, то при поднятии сможет задетектить проблему - никакие операции из транзакции не применятся.

изоляция получается “из коробки“ благодаря однопоточности.

Еще есть возможность делать optimistic locking с помощью команды WATCH. Клиент говорит Redis-у «я сейчас получу от тебя значения по некоторым ключам, проверю у себя кое-что в бизнес-логике и пришлю тебе команды, которые ты выполнишь только если те ключи не менялись все это время». Например, пересылка денег от user1 к user2:р быстрые (десятки микросекунд), а потому нет смысла вносить параллелизм, который всё только усложнит.

Вывод: можно реализовать любую бизнес-логику такими транзакциями.

Хранение данных

Redis все держит in memory, но умеет делать дампы. Создание дампа не блокирует операции записи/чтения. Дамп делается раз в какое-то время, а потому можно потерять данные, которые были записаны после последнего дампа. Чтобы этого не случилось, Redis поддерживает еще write ahead log. Но его разумно флашить на диск максимум раз в 1 секунду, иначе обещанная производительность Redis-а убьется. Поэтому нужно смириться с тем, что Redis может потерять данные за последнюю секунду.

**Репликация**

Максимально простецкая single leader репликация. Асинхронная по дефолту. Можно сделать синхронной (команда WAIT), но даже в этом случае до строгой консистентности (как в raft и paxos), как до луны. Например, если мастер упал, то новым может стать реплика с отстающим логом операций 🤦🏻‍♂️ Т.е. потерять данные в кэше элементарно.
Из-за этого Redis ну точно не подходит как primary storage для критических данных.
Хорошо хоть такая репликация и выборы мастера в случае падения полностью автоматизированы.

**Шардирование** 

Тоже довольно простецкое. В Redis всегда 16384 партиции. Каждый пользовательский key попадает в одну партицию через hash(key) mod 16384. 
Когда вы создаете кластер редиса, партиции распределяются по серверам и прибиты гвоздями. Никакой автоматической ребалансировки в зависимости от нагрузки по партициям. Никакого разделения партиции на две, если она стала тяжелее остальных (как это делается во многих бд). Проще говоря, партиции не делятся, не мержатся, не переезжают между серверами. 
Если вы добавляете новый сервер, то вручную вызываете команду reshard, чтобы часть партиций мигрировала на новый сервер. Миграцию Redis делает автоматически.
Информация, на каком сервере живет та или иная партиция, хранится на каждом сервере.

Разработчик может назначать hash-tag каждому ключу, что привяжет ключ к определенной партиции. Полезно, когда хочется иметь группу ключей в одной партиции - это позволяет делать транзакции над этой группой ключей.
Транзакции же над несколькими партициями невозможны (2-phase commit не делали и вряд ли сделают).

Встроенные структуры данных
По ключу можно хранить не просто строку/число, но и всем знакомые структуры данных: массив, hash map со своими key value, tree map со своими key value и сортировкой по ключам.
Такие штуки позволяют поддерживать более сложные сценарии, чем обычный key value. Например, если бы вы делали инстаграм, то могли бы хранить фид для пользователей в виде userId -> array of post ids. Могли бы добавлять новые посты в конец списка, иметь всю эту логику в кэше.

Вывод

В целом в Redis чувствуется следующая философия: сделать все аспекты хранилища максимально просто, ведь для большинства кэш-сценариев этого более чем достаточно.

Redis полноценен и для large scale сценариев: тот же фид инстаграма с их ~500M DAU вполне бы влез в шардированный кластер Redis-a даже с его ограничением в 16384 партиции.

Интересно будет сравнить его с конкурентами (Memcached, Tarantool).
